# API-контракт: Сканер ↔ Бэкенд ↔ Стена

**Версия:** v1 (без формального версионирования — breaking-изменения см. раздел
[Versioning](#versioning--совместимость) внизу). Актуально на 2026-07-08,
синхронизировано с кодом (`app.js`, `backend/main.py`, `index.html`).

Это единый источник правды для всех трёх узлов архитектуры. `backend/README.md`
не должен ему противоречить — если что-то меняется здесь, обновляем и там.

## Компоненты

- **Сканер** (`index.html` + `app.js`, открывается на телефоне) — раскраска на
  экране ИЛИ фото рисунка с бумаги, отправка на бэкенд.
- **Бэкенд** (`backend/main.py`, FastAPI) — приём, обработка изображения,
  раздача рыбок подписчикам.
- **Стена** — потребитель `WS /ws/wall`. **`aquarium.html` — единственный
  production Wall-клиент, показывается на демо.** У него boid-флокинг,
  механика кормления и полноценный HTTP-fallback с подстраховкой при обрыве
  WS (см. раздел 3.2). Fallback — `GET /api/fish/latest`.
  - Встроенный экран `#screen-aquarium` в `index.html`/`app.js` (тот же
    сайт, что и Сканер) — **dev/test-инструмент**, чтобы проверять Сканер
    без второго физического устройства. Он проще (нет кормления, нет
    boid-флокинга, HTTP-fallback только разовый при загрузке, см. раздел
    3.1) и **не предназначен для показа на демо**.

  Оба слушают `WS /ws/wall` и корректно дедуплицируют рыб по `fish_id` при
  реконнекте. Любой будущий клиент Стены (на любом стеке — тот же Canvas 2D,
  WebGL-движок и т.п.) должен реализовать тот же контракт независимо, поверх
  `WS /ws/wall` / `GET /api/fish/queue` / `GET /api/fish/latest` /
  `GET /api/fish/{id}/image`.

---

## Соглашение об именовании полей

Стиль **смешанный, и это осознанно, а не баг**:

- Тело запроса `POST /api/fish` — **camelCase** (`fishType`, `createdAt`) —
  так исторически шлёт `app.js`.
- Сообщения `WS /ws/wall` и ответ `GET /api/fish/queue` — **snake_case**
  (`fish_id`, `created_at`, `image_url`) — так отдаёт FastAPI/Python, и так же
  их читает встроенный аквариум в `app.js` (`meta.fish_id`, `meta.image_url`).

⚠️ Не переименовывайте поля в одном месте без синхронной правки всех
клиентов (`app.js` — и как отправитель, и как читатель WS; `backend/main.py`;
`aquarium.html`; любой будущий клиент Стены). Оба действующих Canvas
2D-клиента (встроенный аквариум и `aquarium.html`) уже завязаны на текущие
snake_case-имена в WS (`meta.fish_id`, `meta.image_url`) — их смена без
причины — риск сломать рабочий код перед демо.

## HTTP endpoints

| Метод | Путь | Назначение |
|---|---|---|
| `POST` | `/api/fish` | Сканер → Бэкенд: отправка готовой рыбки |
| `GET` | `/api/fish/{fish_id}/image` | Скачать PNG рыбки по `image_url` из `new_fish` |
| `GET` | `/api/fish/queue?since=<unix_ts>` | Fallback-поллинг без WebSocket — массив рыб, используется встроенным аквариумом |
| `GET` | `/api/fish/latest` | Fallback-поллинг без WebSocket — алиас, один последний объект рыбки (не массив), используется `aquarium.html`. См. раздел 3 |
| `WS` | `/ws/wall` | Бэкенд → Стена: живая трансляция новых рыб |
| `GET` | `/api/health` | Liveness-проверка (`{"status":"ok","time":...,"queue_len":N}`) — можно использовать для кнопки «Проверить соединение» на экране настроек Сканера |
| `GET` | `/`, `/app.js`, `/aquarium.html` | Отдача статики — Сканер и встроенный аквариум это `index.html`+`app.js`, отдельный Canvas 2D-клиент Стены — `aquarium.html` (нужно для деплоя, когда всё живёт на одном порту/хосте) |

Адрес сервера настраивается в самом приложении Сканера (поле на главном
экране) — менять код не нужно, просто вписать реальный `http(s)://<ip>:<port>`.

## 1. Сканер → Бэкенд: `POST /api/fish`

```
POST /api/fish
Content-Type: application/json
```

### Тело запроса

```json
{
  "fishType": "clownfish",
  "mode": "draw",
  "image": "data:image/png;base64,iVBORw0KGgo...",
  "createdAt": "2026-07-08T06:07:56.318Z"
}
```

| Поле | Тип | Значения / описание |
|---|---|---|
| `fishType` | string | `clownfish`, `shark`, `octopus`, `whale`, `squid`, `crab`, `dolphin`, `turtle`, `fish`. Сервер **не валидирует** значение — принимает любую строку (см. [Versioning](#versioning--совместимость)). |
| `mode` | string | `draw` (раскраска на экране) или `scan` (фото с камеры) |
| `image` | string | base64 data URL. **Формат зависит от `mode`, см. ниже** |
| `createdAt` | string | ISO 8601 timestamp, опционально |

### Формат `image` по режимам — кто убирает фон

Это принципиальный момент архитектуры, ранее не был чётко зафиксирован и
противоречил сам себе (контракт говорил и «PNG всегда», и «сканер шлёт JPEG»
одновременно). Решение:

- **`mode = "draw"`** — рисунок сделан на HTML canvas в приложении Сканера.
  Клиент шлёт **PNG**, фон уже прозрачный по построению (canvas создаётся
  прозрачным). Бэкенд (`_process_canvas`) только нормализует размер до
  512×512 — дополнительной обработки не требуется.

- **`mode = "scan"`** — фото рисунка на бумаге с камеры телефона. Клиент шлёт
  **сырое, необработанное фото целиком, JPEG**, БЕЗ клиентской обрезки и БЕЗ
  клиентского удаления фона.

  **Почему именно так, а не наоборот:** серверный пайплайн
  (`backend/image_processing.py::_process_photo`) сначала ищет сам лист
  бумаги как четырёхугольник в кадре (`_extract_paper`, нужен видимый край
  листа на фоне стола), выпрямляет перспективу, и только потом по
  напечатанному контуру рыбки отделяет её от руки/ручки/края листа
  (`_select_fish_mask`, скоринг по площади/центрированности/выпуклости).
  Если на вход придёт уже обрезанная по контуру рисунка и уже прозрачная
  картинка — у сервера не останется ни края листа для поиска
  четырёхугольника, ни красок вне контура рыбки для правильной сегментации,
  и его алгоритм либо сработает непредсказуемо, либо просто откатится на
  грубый fallback (`_fallback_biggest_blob`). Проверено на синтетике
  (`stress_test_photos.py`), но не на «уже пожёванном» клиентом фото — это
  ровно тот сценарий, который ломается на реальном демо, а не на тестах.

  Поэтому: **бэкенд — единственный владелец удаления фона и вырезания рыбки
  для `mode=scan`**. Клиент отвечает только за захват кадра и (опционально)
  показ ребёнку локально обработанного превью — это **не то, что уходит на
  сервер** (см. правку в `app.js`, раздел ниже).

### Ответ сервера

Успех — HTTP 200:
```json
{ "ok": true, "fishType": "clownfish", "mode": "draw" }
```

Ошибка — HTTP 400/500:
```json
{ "ok": false, "error": "текст ошибки" }
```
Сканер (текущая реализация `sendFish` в `app.js`) проверяет **только
HTTP-статус** (`res.ok`), тело `{ok:false,...}` на данный момент не парсится
для принятия решений — источник истины для UI «Попробовать снова» это
HTTP-код, а не тело ответа. При ошибке показывается кнопка «Попробовать
снова», по клику — повтор того же запроса с тем же `imageDataUrl` (без
повторного захвата кадра).

### CORS

Настроено через `CORSMiddleware` в `backend/main.py`:
```
allow_origins=["*"]
allow_methods=["POST", "OPTIONS", "GET"]
allow_headers=["Content-Type"]
```
Фактический ответ на `OPTIONS`-preflight (проверено живым запросом,
`Access-Control-Request-Headers: Content-Type`) — это то, что реально
отдаёт Starlette `CORSMiddleware`, а не буквальная конфигурация выше:
```
HTTP/1.1 200 OK
access-control-allow-origin: *
access-control-allow-methods: POST, OPTIONS, GET
access-control-allow-headers: Accept, Accept-Language, Content-Language, Content-Type
```
Два нюанса, которые раньше были описаны неточно:
- **Код ответа на preflight — `200`, не `204`.** Для браузеров это не имеет
  значения (принимают любой `2xx`), но если тестируете руками curl'ом и
  ждёте `204` — не ждите.
- **`Access-Control-Allow-Headers` в ответе шире, чем сконфигурировано** —
  Starlette добавляет «safelisted» заголовки (`Accept`, `Accept-Language`,
  `Content-Language`) к тому, что указано в `allow_headers`. Функционально
  не проблема (нужный `Content-Type` есть), просто список заголовков не
  равен буквально `allow_headers=["Content-Type"]`.

### Размер запроса

- `mode=draw`: PNG с canvas, обычно единицы–десятки КБ.
- `mode=scan` (после фикса — сырой JPEG с камеры, без клиентского сжатия
  под транспарентность): может быть от сотен КБ до нескольких МБ в
  зависимости от разрешения камеры и `toDataURL('image/jpeg', quality)`.
- **Рекомендуемый предел — 20 МБ**, с запасом на некачественно сжатые/крупные
  кадры. Это **не enforced лимит сервера** — ни в `backend/main.py`, ни в
  запуске uvicorn нет никакой валидации размера body, FastAPI/Starlette по
  умолчанию тоже не ограничивают. Число носит рекомендательный характер для
  клиента (не слать заведомо больше) и как ориентир для будущей настройки
  реального лимита, если понадобится защита от аномально больших/вредоносных
  запросов.

## 2. Бэкенд → Стена: `WS /ws/wall`

Основной канал получения новых рыб в реальном времени.

### Подключение и начальная синхронизация

При установке соединения сервер сразу шлёт **последние 20 рыб** из истории
(`fish_queue[-20:]`), каждую отдельным сообщением — это и есть встроенный
механизм «не терять данные при реконнекте».

⚠️ Следствие: **при каждом реконнекте эти 20 рыб прилетают заново**. Клиент
обязан дедуплицировать по `fish_id` самостоятельно — сервер не помнит, что
конкретный клиент уже видел. Оба действующих референса делают это правильно
(`app.js`: `aqua.seen` — `Set<fish_id>`; `aquarium.html`: `seen` — тот же
паттерн), `addFish()` в обоих — ранний `return`, если id уже видели. Любой
новый клиент Стены обязан повторить эту же логику.

### Формат сообщения

```json
{
  "type": "new_fish",
  "fish_id": "15c8ee80",
  "fishType": "clownfish",
  "mode": "draw",
  "created_at": 1783491767.06,
  "avg_color": [254, 142, 25],
  "width": 512,
  "height": 512,
  "image_url": "/api/fish/15c8ee80/image"
}
```

| Поле | Тип | Описание |
|---|---|---|
| `type` | string | Сейчас только `"new_fish"`. Клиент обязан игнорировать неизвестные `type` (а не падать) — задел на будущие типы событий. |
| `fish_id` | string | Уникальный id, ключ дедупликации |
| `fishType` | string | Как прислал Сканер, без валидации |
| `mode` | string | `draw` \| `scan` |
| `created_at` | number | Unix timestamp (секунды, float) — **не ISO-строка**, в отличие от `createdAt` в запросе |
| `avg_color` | [r,g,b] | Средний цвет рыбки, опционально для доп. эффектов (подсветка, пузырьки) |
| `width`, `height` | number | Всегда 512×512 на сегодня |
| `image_url` | string | **Относительный** путь. Полный URL = `origin_бэкенда + image_url`. Картинка качается отдельным `GET`, в сам WS-payload не встраивается |

`image_url` → `GET /api/fish/{fish_id}/image` → `200 image/png` (RGBA,
512×512, рыбка вписана по центру с сохранением пропорций, фон прозрачный) или
`404 {"error":"not_found"}`.

### Keepalive / heartbeat

Сервер сейчас **не шлёт пингов сам** — держит соединение открытым в цикле
`await websocket.receive_text()`, ожидая любых входящих сообщений (payload
игнорируется). Он не обнаруживает «подвисшее» соединение самостоятельно.
Клиенту рекомендуется периодически слать любой текст (например `"ping"`)
каждые ~20-30 сек, чтобы транспортные прокси/файрволы не рвали простаивающий
сокет молча.

### Reconnect-политика клиента

Не зафиксирована формально — оба клиента переподключаются с фиксированной
задержкой без backoff, но с **разными** интервалами: dev-инструмент `app.js`
— через **1.5 сек** после `onclose`/`onerror`, production-клиент
`aquarium.html` — через **3 сек** после `onclose` (см. `connectWS()`). Для
хакатона расхождение не критично, но если какому-то клиенту Стены нужен
конкретный интервал — согласовать отдельно, это не часть строгого контракта.

В `aquarium.html` (единственном production-клиенте) reconnect дополнительно
считает подряд идущие неудачи и после 3 (`HTTP_FALLBACK_AFTER`) включает
HTTP-поллинг `/api/fish/latest` как подстраховку, не прекращая пытаться
восстановить WS — см. раздел 3.2. Это не «переключение» на другой канал, а
временное дублирование на время, пока WS не восстановится.

## 3. Fallback без WebSocket

Два эндпоинта на выбор — оба реально используются действующими референсами,
оба нужно поддерживать.

### 3.1. `GET /api/fish/queue` (канонический, используется dev-инструментом `app.js`)

```
GET /api/fish/queue?since=<unix_timestamp>
```

Возвращает **массив** объектов в том же формате, что и WS `new_fish`
(без обёртки `type`), только `created_at > since`.

Использование:
- При старте (или если WS недоступен) — запросить `since=0`, получить всю
  историю, дедуплицировать по `fish_id` (тот же набор, что и для WS).
- Встроенный аквариум в `app.js` (dev/test-инструмент для Сканера, **не
  production-Стена** — см. «Компоненты») сейчас делает **разовый** запрос
  при инициализации, безусловно, независимо от статуса WS — это не
  поллинг. Если WS-соединение не поднимется вообще (не только на старте,
  но и все retry подряд), новых рыб клиент не увидит, пока не перезагрузит
  страницу. Для dev-инструмента это осознанно принято как есть — полноценный
  fallback с подстраховкой реализован в production-клиенте `aquarium.html`
  (раздел 3.2), дублировать его в `app.js` не требуется.

### 3.2. `GET /api/fish/latest` (алиас, используется production-клиентом `aquarium.html`)

```
GET /api/fish/latest
```

Возвращает **один объект** — последнюю добавленную рыбку в том же формате,
что и элемент `/api/fish/queue` (без обёртки `type`), или `{}` (пустой
объект, без `fish_id`), если рыб ещё не было. Это тонкий алиас над тем же
`fish_queue` на сервере — не отдельный источник данных, просто другая форма
ответа (объект вместо массива) для клиентов, которым не нужна вся история,
а только «есть ли что-то новое прямо сейчас».

`aquarium.html`, поллинг раз в `CFG.pollMs` в `startHttpPoll()`, сравнивает
`data.fish_id` с локально запомненным `lastFishId` — если отличается,
считает рыбку новой. Заметьте: у этого способа есть предел — если между
двумя опросами добавили **больше одной** рыбки, клиент увидит только самую
последнюю и пропустит промежуточные (в отличие от `/api/fish/queue`, который
отдаёт весь диапазон). Для одиночного демо-показа это не критично, но при
частой одновременной отправке рыб с нескольких Сканеров — стоит перейти на
`/api/fish/queue`.

**Когда включается `startHttpPoll()`:** не только на синхронную ошибку
конструктора `new WebSocket(...)` (редкий случай), но и — начиная с этой
версии — после **3 неудачных `onclose` подряд** в `connectWS()` (~9 сек при
задержке реконнекта 3с). До этого фикса реальный обрыв связи (упавший
бэкенд, не та Wi-Fi сеть) приводил только к бесконечному WS-реконнекту без
HTTP-подстраховки — HTTP-поллинг был по факту недостижим при самом
вероятном сценарии сбоя на демо. HTTP-поллинг работает **параллельно** с
продолжающимися попытками WS-реконнекта (не вместо них) и автоматически
останавливается (`clearInterval`) в `ws.onopen`, как только WS восстановится
— во избежание задваивания источника рыб.

Эта ручка была добавлена в бэкенд задним числом, чтобы привести уже рабочий
`aquarium.html` в соответствие с контрактом с минимальными изменениями (не
трогая его отлаженную логику поллинга) — см. «Известные проблемы» внизу.

## 4. Ошибки и коды

| Код | Когда | Тело |
|---|---|---|
| `200` | Успех `/api/fish` | `{"ok":true,"fishType":...,"mode":...}` |
| `400` | Битый data URL (`bad_image_data`) или ошибка обработки изображения (`processing_failed`, включая «рыбку на листе не нашли») | `{"ok":false,"error":"..."}` |
| `404` | `GET /api/fish/{id}/image` для несуществующего id | `{"error":"not_found"}` |
| `422` | Тело запроса не соответствует схеме (Pydantic-валидация FastAPI, например отсутствует обязательное поле) | стандартный FastAPI validation error |
| `204` | `OPTIONS` preflight | — |

Ошибка `processing_failed` на `mode=scan` — самый вероятный сбой на демо
(неудачное освещение/ракурс → сервер не нашёл рыбку на листе). Строка `error`
у обеих кодовых ошибок — это **префикс + `: ` + текст исключения**
(`f"processing_failed: {e}"` / `f"bad_image_data: {e}"` в `backend/main.py`),
а не голый код — сверяйте по `startsWith(...)`, не по точному равенству.

**Клиент обязан парсить тело ответа**, а не только HTTP-статус, чтобы решить,
какую кнопку показать (реализовано в `sendFish`/`showSendError`, `app.js`):

- HTTP не-2xx / сеть недоступна / тело не парсится как JSON → значит проблема
  сетевая/серверная, повтор того же кадра имеет смысл → кнопка «Попробовать
  снова» (повторяет тот же POST с тем же `imageDataUrl`).
- `error` начинается с `processing_failed` или `bad_image_data` **и**
  `mode === "scan"` → проблема в самом кадре, повтор ничего не изменит →
  кнопка «Переснять фото» (тот же флоу, что `retakeBtn`: возврат на экран
  камеры + `initCamera()`, не повтор запроса). Текст ошибки при этом
  человеческий, не технический («Не видим рыбку на листе. Попробуй переснять
  при хорошем свете» и т.п.).
- Для `mode === "draw"` эти же коды ошибок (маловероятны, но возможны на
  `bad_image_data`, если, например, canvas дал битый data URL) намеренно
  **не** ведут на «Переснять фото» — камеры в draw-режиме нет, ведут на
  экран с раскраской. Пока для этого случая используется тот же fallback,
  что и для сетевых ошибок («Попробовать снова», хотя технически повтор с
  тем же битым data URL не поможет) — если такое всплывёт на практике, нужен
  отдельный флоу «вернуться к раскраске», сейчас не реализован (не было в
  задаче).

Ручной чек-лист проверки поведения — [SCAN_ERROR_UX_CHECKLIST.md](SCAN_ERROR_UX_CHECKLIST.md).

## 5. Versioning / совместимость

- Формального версионирования нет (нет `/v1/` в путях, нет заголовка
  версии). Для 24-часового хакатона — осознанное решение, не пробел.
- Если понадобится breaking-изменение формата (например, смена структуры
  `new_fish`) — вводить новый путь (`/ws/wall/v2`) или новое поле `type`,
  не ломая существующих подписчиков молча.
- `fishType` не валидируется сервером. Если Сканер пришлёт значение вне
  списка из 9 — бэкенд всё равно примет и разошлёт как есть. **Любой клиент
  Стены должен сам показывать визуальный fallback на неизвестный
  `fishType`** (дефолтная рыбка/спрайт вместо конкретной формы), а не
  падать/пропускать рыбку. Ни встроенный аквариум, ни `aquarium.html` сейчас
  этого не делают явно — оба просто грузят `image_url` как картинку
  независимо от `fishType`, так что до появления невалидного значения это не
  проявится, но полагаться на это не стоит.
- `mode` кроме `draw`/`scan` сервером не проверяется (`proc_mode = "canvas"
  if mode == "draw" else "photo"` — любое не-`"draw"` значение трактуется
  как фото). Если в будущем появится третий режим — здесь будет тихий баг.
- Открытый вопрос не по протоколу, а по продукту: механика «кормления
  рыбок» (см. ТЗ) пока не имеет ни одного события в этом контракте — ни
  HTTP, ни WS. Если кормление предполагает связь между устройствами
  (действие на Сканере/сайте влияет на Стену), нужен отдельный
  event-контракт до начала реализации этой части.
- **Known limitation:** `mode=draw` + `bad_image_data` не показывает кнопку
  «Переснять» (нет камеры в draw-режиме), падает на общий retry
  («Попробовать снова» с тем же битым data URL — реально не поможет, но
  других вариантов сейчас нет). Решение отложено, риск принят тимлидом
  2026-07-08. Реализация — см. `showSendError` в `app.js`.

## Известные проблемы в старой версии контракта (исправлено здесь)

Для истории — что было не так в предыдущей редакции этого файла:

1. Противоречие «PNG всегда» vs «сканер шлёт JPEG в scan» — разрешено:
   `draw` → PNG (клиент), `scan` → JPEG сырое фото (клиент), удаление фона
   — только на сервере.
2. WS-канал, `GET image`, `GET queue`, `GET health` отсутствовали в
   контракте целиком (были только в `backend/README.md`) — добавлены.
3. Ссылка на `../animagia-mock-server/server.js` вела в несуществующую
   директорию — такой директории нет в репозитории. Рабочий референс для
   локальной проверки: `backend/main.py` (сам сервер) +
   `backend/simulate_scanner.py` (эмулятор Сканера для curl-less теста).
4. Дедупликация по `fish_id` при реконнекте WS не была специфицирована —
   добавлена как обязательное требование к клиенту.
5. После удаления отдельного `aquarium.html` (слит в единый сайт) в
   `backend/main.py` остался мёртвый роут `GET /aquarium.html`,
   отдающий несуществующий файл — удалён. **С тех пор устарело:** `aquarium.html`
   вернулся в репозиторий отдельным файлом (новый Canvas 2D-клиент Стены,
   869 строк, свой рендер-луп и HTTP-fallback) — роут `GET /aquarium.html`
   восстановлен в `backend/main.py`, см. пункт 6.
6. Команда решила остаться на Canvas 2D вместо перехода на Three.js —
   упоминания Three.js как целевого движка Стены убраны из документации и
   комментариев, формулировки переведены на нейтральные («клиент Стены»
   без привязки к конкретному рендер-стеку). Заодно обнаружено и устранено:
   - `aquarium.html` опрашивал `GET /api/fish/latest`, которого не было в
     бэкенде (HTTP-fallback был на 100% нерабочим — 404 на каждый запрос,
     тихо проглатывался). Добавлен как официальный алиас-эндпоинт
     (см. раздел 3.2), без изменений в уже отлаженном клиентском коде.
   - Гипотеза о рассинхроне полей (`meta.fishid`/`meta.imageurl` без
     подчёркивания) **не подтвердилась** — проверено построчно, `aquarium.html`
     везде использует корректный `fish_id`/`image_url`, совпадающий с
     контрактом. Багов с полями не найдено.
7. `aquarium.html` зафиксирован как единственный production Wall-клиент
   для демо, встроенный аквариум в `app.js` — dev/test-инструмент для
   Сканера (см. «Компоненты»). По итогам полного аудита репозитория
   исправлено в `aquarium.html`:
   - HTTP-fallback (`/api/fish/latest`) был фактически недостижим при
     реальном обрыве связи — `startHttpPoll()` вызывался только на
     синхронную ошибку конструктора `new WebSocket(...)`, а не на
     `onclose`/`onerror`. Добавлен счётчик неудачных попыток реконнекта:
     после 3 подряд включается HTTP-поллинг параллельно с продолжающимся
     WS-реконнектом, и останавливается в `onopen` (см. раздел 3.2).
   - WS-обработчик не проверял `data.type` перед `addFish()` — добавлена
     проверка `data.type === 'new_fish'`, как того требует контракт
     («клиент обязан игнорировать неизвестные `type`»).
   - Уточнена документация CORS (реальный код preflight — `200`, не `204`;
     реальный список `Allow-Headers` шире сконфигурированного) и лимита
     body (20 МБ — рекомендация клиенту, не enforced лимит сервера).
