# Анимагия — Backend (узел «Сканер → Сервер → Стена»)

Архитектура: **Клиент-Сервер в локальной сети**. Сервер запускается на одном
ноутбуке в той же Wi-Fi/локальной сети, что и телефон-Сканер и ноутбук со
«Стеной». **`../aquarium.html` — единственный production Wall-клиент,
показывается на демо.** Встроенный аквариум в `../index.html`/`../app.js` —
dev/test-инструмент для проверки Сканера без второго устройства, на демо не
показывается. Подробности — [`../API_CONTRACT.md`](../API_CONTRACT.md#компоненты).

## Как запустить одной командой

Из корня репозитория (не из `backend/`!):

```bash
# Linux / macOS
./start.sh
```
```bat
:: Windows
start.bat
```

Оба скрипта делают то же самое, что раньше приходилось делать руками: `cd
backend`, активация venv (если есть), `uvicorn main:app --host 0.0.0.0
--port 3000`. Если раньше ловили `ModuleNotFoundError` из-за запуска не из
той папки — дело было именно в этом, скрипты сами переходят в `backend/`
перед запуском.

Зависимости они не ставят — это одноразовый шаг, до первого запуска:
```bash
cd backend && pip install -r requirements.txt --break-system-packages
```

**После запуска — отдельный шаг, скрипт этого не делает:** нужно узнать IP
этого компьютера в локальной сети, чтобы с телефона подключить Сканер:
- Windows: `ipconfig` (смотреть `IPv4-адрес` активного адаптера)
- Linux/Mac: `ip addr` или `ifconfig`

Сканер и Стена (`aquarium.html`) обращаются по адресу
`http://<этот-IP>:3000/`.

Проверка, что сервер жив: `GET http://<IP>:3000/api/health`

### Запуск вручную (то же самое, без скрипта)

```bash
cd backend
pip install -r requirements.txt --break-system-packages
uvicorn main:app --host 0.0.0.0 --port 3000
```

### Проверено на по-настоящему чистом окружении — что может пойти не так

Прогнано на свежем `git clone` в отдельную папку, с venv, изолированным от
любых ранее установленных глобально пакетов (иначе тест ничего не проверяет
— именно так выглядела предыдущая, недостаточно строгая проверка).

- **Совсем нет `backend/.venv` и зависимости нигде не стоят.** Скрипт не
  падает молча и не виснет — печатает:
  ```
  venv не найден. Создайте его и поставьте зависимости (один раз):
    python -m venv backend/.venv
    source backend/.venv/bin/activate
    pip install -r backend/requirements.txt
  ```
  и выходит с кодом `1`. (Windows: тот же текст в `start.bat`, путь к
  активации — `backend\.venv\Scripts\activate`.)
- **`backend/.venv` существует, но `uvicorn` в нём не установлен** (папка
  создана, `pip install` не выполнялся, либо venv битый/от другого
  Python) — сообщение отличается от предыдущего, явно говорит, что venv
  найден, но зависимостей в нём нет, и предлагает тот же `pip install`
  без пересоздания venv:
  ```
  backend/.venv найден, но uvicorn в нём не установлен (зависимости не
  ставились или venv битый). Выполните:
    source backend/.venv/bin/activate
    pip install -r backend/requirements.txt
  ```
- **Порт 3000 уже занят** (не завершился прошлый `uvicorn`, или порт
  занят чем-то ещё) — скрипт не виснет, `uvicorn` сам обрывается с явной
  ошибкой на старте:
  ```
  ERROR: [Errno 10048] error while attempting to bind on address
  ('0.0.0.0', 3000): обычно разрешается только одно использование адреса
  сокета (протокол/сетевой адрес/порт)
  ```
  и процесс завершается (ненулевой код выхода). Технический текст ошибки
  на демо может смутить — по-человечески это означает: **бэкенд уже
  где-то запущен на этом порту, или процесс не до конца завершился в
  прошлый раз.** Что делать: проверить, не открыт ли уже
  `http://localhost:3000` в браузере (возможно, всё уже работает и
  перезапуск не нужен); если нет — найти и остановить старый процесс
  (Windows: `Get-Process -Id (Get-NetTCPConnection -LocalPort 3000).OwningProcess | Stop-Process`
  в PowerShell; Linux/Mac: `lsof -i :3000` → `kill <PID>`), либо
  временно запустить на другом порту (`uvicorn main:app --port 3001`) —
  адрес в UI Сканера/Стены нигде не настраивается вручную, он всегда
  берётся из origin страницы: просто откройте
  `http://<этот-IP>:3001/` вместо `:3000`, ничего в интерфейсе
  донастраивать не нужно.
- **Полный успешный прогон с нуля** (чистый venv → `pip install` →
  `start.sh`) проверен целиком: сервер поднимается, `GET /api/health`
  отвечает `200`.

## API-контракт

### 1. Сканер → Сервер: отправка рыбки

Реализовано строго по [`../API_CONTRACT.md`](../API_CONTRACT.md), который уже
согласован с устройством «Сканер» (`../index.html` + `../app.js`):

```
POST /api/fish
Content-Type: application/json

{
  "fishType": "clownfish",
  "mode": "draw",   // "draw" (раскраска на экране) | "scan" (фото с камеры)
  "image": "data:image/png;base64,....",
  "createdAt": "2026-07-08T06:07:56.318Z"
}
```

Ответ:
```json
{ "ok": true, "fishType": "clownfish", "mode": "draw" }
```
`mode=draw` обрабатывается как `canvas` (фон уже прозрачный), `mode=scan` —
как `photo` (сервер сам находит лист и вырезает рыбку, см. ниже).

### 2. Сервер → Стена: получение готовой рыбки (WebSocket, основной канал)

```
WS /ws/wall
```

При подключении сразу присылает последние ~20 рыб (на случай, если клиент
Стены переподключился). Дальше — push новых рыб в реальном времени:

```json
{
  "type": "new_fish",
  "fish_id": "15c8ee80",
  "fishType": "clownfish",
  "mode": "draw",
  "created_at": 1783491767.06,
  "avg_color": [254, 142, 25],
  "width": 512,
  "height": 512,
  "image_url": "/api/fish/15c8ee80/image"
}
```

`image_url` — относительный путь, картинку нужно скачать отдельным GET
запросом: `GET http://<IP>:3000/api/fish/{fish_id}/image` → PNG с
прозрачным фоном, готовый под спрайт/текстуру рыбки в клиенте Стены.

`avg_color` — средний цвет рыбки, можно использовать для доп. эффектов
(подсветка, частицы, цвет пузырьков и т.п.), это опционально.

### 3. Fallback без WebSocket (если Wi-Fi/сокеты подведут на демо)

```
GET /api/fish/queue?since=<unix_timestamp>
```

Клиент Стены может поллить этот эндпоинт каждые 1-2 секунды и сравнивать с
локально известными `fish_id`, чтобы не проигрывать анимацию повторно.
Возвращает то же самое, что приходит через WebSocket, только массивом.
Используется встроенным аквариумом в `../app.js`.

Есть и альтернативный эндпоинт для клиентов, которым проще опрашивать «есть
ли что-то новое» одним объектом, а не массивом:

```
GET /api/fish/latest
```

Возвращает последнюю рыбку (или `{}`, если рыб ещё не было) — тот же формат
полей, что и элемент `/api/fish/queue`. Используется `../aquarium.html`.
Подробности и ограничения — см. [`../API_CONTRACT.md`](../API_CONTRACT.md),
раздел 3.2.

## Формат картинки, которую получает Стена

- PNG, RGBA, всегда 512x512
- Рыбка вписана в центр с сохранением пропорций, фон полностью прозрачный
- Никакой дополнительной обработки на клиенте не требуется — можно грузить
  прямо как обычное изображение (`new Image()` + `img.src`, как делают оба
  действующих референса) и рисовать на прозрачном фоне без доп. масок.

## Как устроена обработка (image_processing.py)

- **mode=canvas** (рисунок в вебе/приложении на Сканере): просто нормализуем
  размер до 512×512, фон уже прозрачный.
- **mode=photo** (фото рыбки на бумаге):
  1. Находим сам лист бумаги в кадре (детектор четырёхугольника, как в
     сканерах документов) и выпрямляем перспективу — так убираем стол,
     тени и угол съёмки.
  2. Находим рыбку по её **напечатанному тёмному контуру** (`_select_fish_mask`):
     адаптивный порог по тёмным линиям (устойчив к неровному свету), замыкаем
     контур в кольцо и из всех фигур выбираем рыбку скорингом «крупная +
     ближе к центру + сплошная/выпуклая + не липнет к краям кадра».
     Так в результат попадает только рыбка, а **рука** (она не тёмная — в
     кандидаты не попадает), **ручка/карандаш** (тонкие — отсекаются по
     площади) и **край листа** (липнет ко всем сторонам) — отбрасываются.
  3. Заливаем выбранный контур целиком (внутренности рыбки — глаз, блики —
     сохраняются, фон внутри не выгрызается), обрезаем, вписываем в 512×512,
     отдаём PNG с альфой.
- Если контур рыбки найти не удалось — fallback `_fallback_biggest_blob`
  (старое поведение «самое большое цветное/тёмное пятно»), чтобы сканер не
  падал в ошибку на нестандартном фото.
- Проверки: `python3 test_scan_intrusions.py` (рыбка + рука + ручка → только
  рыбка) и `python3 stress_test_photos.py` (свет/тень/блик/наклон/край).

## Разбивка на 2 человек (backend)

**Человек A — API/инфраструктура**
- Эндпоинты `/api/fish`, `/api/fish/queue`, `/api/fish/{id}/image`
- WebSocket `/ws/wall`, рассылка и переподключение
- CORS, тестовые curl/Postman-запросы, тестовый HTML-клиент-эмулятор
  сканера (чтобы команда Стены могла тестировать без реального телефона)
- Интеграция итогового IP-адреса/порта с командами Сканера и Стены

**Человек B — обработка изображений**
- `_extract_paper` / `_extract_drawing_mask` — подгонка порогов под
  реальные условия освещения на месте хакатона (важно протестировать на
  реальных фото, не только на синтетике!)
- Обработка edge-кейсов: неровный свет, тень от руки, блик от вспышки, лист
  под углом, рыбка вылезает за край листа — все эти случаи прогоняются
  синтетически через `python3 stress_test_photos.py` (см. вывод со статусом
  OK/SUSPICIOUS/FAIL по каждому кейсу + сохранённые `stress_*_input/output`
  картинки для визуальной проверки). Синтетика — не замена реальным фото,
  но ловит явные регрессии в `_extract_paper`/`_extract_drawing_mask` без
  необходимости каждый раз фотографировать заново.
- Замер скорости обработки (должно быть <1-2 сек на фото)

## Что нужно синхронизировать с тимлидом/другими группами

1. Финальный порт и то, что сервер должен быть в одной Wi-Fi сети со
   всеми устройствами (важно продумать заранее — вдруг на площадке
   гостевой Wi-Fi блокирует связь между устройствами, тогда нужен свой
   роутер/хотспот).
   ✅ Порт-мисматч, который раньше был здесь (поле адреса Сканера по
   умолчанию ждало `:3000`, а бэкенд по умолчанию поднимался на `:8000`) —
   устранён: `start.sh`/`start.bat` (см. «Как запустить одной командой»)
   и обновлённая ручная команда теперь тоже используют `--port 3000` по
   умолчанию, отдельно донастраивать что-либо не нужно. Поля для ручного
   ввода адреса в UI больше нет вообще (см. `API_CONTRACT.md`) — если
   запускаете бэкенд на другом порту, просто открывайте Сканер по адресу
   с этим портом, адрес подхватится сам из origin страницы.
2. Формат `mode` — если Сканер всегда фотографирует бумагу, `mode=draw`
   можно не поддерживать вообще и упростить.
3. Нужны ли стороне Стены дополнительные метаданные в `new_fish` (например,
   имя ребёнка, номер очереди на анимацию и т.п.) — сейчас есть заготовка
   под доп. поля.
4. Сейчас в репозитории два независимых Canvas 2D-клиента Стены (встроенный
   аквариум в `../app.js` и отдельный `../aquarium.html`), у них разные
   HTTP-fallback эндпоинты и разный интервал реконнекта WS (1.5с / 3с,
   см. `../API_CONTRACT.md`). Стоит решить с командой, нужны ли оба
   постоянно, или один из них — временный/экспериментальный.

## Roadmap: улучшения визуализации Стены

Референсные Canvas 2D-клиенты (встроенный аквариум и `../aquarium.html`)
остаются рабочей production-версией контракта Стены — команда решила не
переходить на отдельный игровой движок. Дальнейшие улучшения — в рамках
того же Canvas 2D / браузерного стека:

- [ ] Вынести Стену в отдельный файл/модуль, не завязанный на код Сканера
      (сейчас есть дублирование между встроенным аквариумом в `../app.js` и
      `../aquarium.html`).
- [ ] Оптимизировать рендер-луп (профилировать на реальном железе стенда —
      количество одновременных рыб, частота кадров).
- [ ] Использовать `image_url` как источник текстуры/спрайта для рыбы —
      уже реализовано в обоих референсах (`new Image()` + `img.src`), но
      стоит унифицировать в один переиспользуемый модуль вместо двух копий.
- [ ] Визуальный fallback (дефолтная рыбка/спрайт) на неизвестный
      `fishType` (см. `../API_CONTRACT.md`, раздел Versioning /
      совместимость) — сейчас не реализован ни в одном из клиентов.

## Быстрый локальный тест без реальных Сканера/Стены

```bash
python3 simulate_scanner.py путь_к_фото.jpg --mode scan --fish-type clownfish
python3 simulate_scanner.py путь_к_рисунка.png --mode draw --fish-type whale

curl http://127.0.0.1:3000/api/fish/queue
```

Или напрямую curl'ом, если под рукой нет файла — JSON по контракту из
раздела 1 выше (поле `image` — data URL с base64).

Проверено на синтетических изображениях — обработка фото и canvas-режим
отрабатывают корректно (см. `test_processing.py`).
